# RLS Audit — Supabase `public` schema

**Date:** 2026-04-18
**Project:** `odvwxgxhacotiuyjyqtk`
**Scope:** every table in the `public` schema — RLS status, policy list, gaps, risks.

This document is a point-in-time snapshot. Re-run the inventory queries in

---

## Summary

- 14 tables total.
- 13 have RLS enabled; **1 does not** (`pre_created_profiles`).
- 44 policies in place across the 13 RLS-enabled tables.
- 2 **critical** issues, 3 **important** gaps, and 4 cleanup items.

---

## Findings by severity

### 🔴 Critical — fix before launch

**1. `pre_created_profiles` has RLS disabled.**
Every row is readable and writable by any authenticated client. If this
table holds PII for people who haven't signed up yet (likely, given the
name), it's an exposure. Action: enable RLS and add policies scoped to
the platform admin (read/write) plus a narrow read policy for whatever
the site needs to expose publicly (if anything).

**2. `venture_submissions` is effectively unprotected.**
All three policies evaluate to `true`. Any authenticated user can read,
insert, or update any row. If submissions contain private pitch info,
founder contact details, or anything not meant to be shared, this leaks.

```
[INSERT] Anyone can insert submissions  CHECK: true
[SELECT] Anyone can read submissions    USING: true
[UPDATE] Anyone can update submissions  USING: true
```

Action: tighten to "submitter can read/update own; admin can read all;
anonymous inserts allowed only if that's intentional."

### 🟡 Should fix — meaningful gaps

**3. `profiles.users_read_own` blocks reading other users' profiles.**
The policy is `USING: (auth.uid() = id)`, meaning a signed-in user can
only see their own row. The mentors directory, workspace member lists,
and introductions UI almost certainly need to read other people's
profiles. If those features work today, they're probably relying on the
admin policy or an RPC bypass. Needs a read policy like
`is_published = true` or `USING: true` depending on what's meant to be
visible.

**4. `workspace_invites` has no DELETE policy.**
Revoking an invite from the UI will silently fail under RLS. Either add
a DELETE policy mirroring the UPDATE one, or handle revocation by
UPDATE + a `revoked_at` column.

**5. Admin identity is hardcoded in 5 policies.**
`agtech_deals` and 4 policies on `introductions` all reference
`auth.email() = 'russellcolevop@gmail.com'`. Risks:

- If your admin email changes, these break.
- If you want a second admin, you need to edit every policy.
- The admin email appears in `pg_policy`, which is queryable by
  anyone with read access to system catalogs.

The production pattern is a SECURITY DEFINER function (`is_admin()`)
that checks a `platform_admins` table, or a custom JWT claim set at
sign-in. The other tables in this project already use `is_platform_admin()`
and `is_admin()` helpers — `agtech_deals` and `introductions` should be
updated to match.

### 🟢 Nice to have — cleanup

**6. Duplicate policies on `eco_program_settings`.**
`[ALL] Managers manage own settings` already covers SELECT, so the
`[SELECT] Managers see own settings` policy is redundant.

**7. Redundant SELECT on `ws_airtable_records`.**
`[ALL] ws_records_write` includes SELECT; the separate `[SELECT]
ws_records_select` policy with a narrower USING is actually broadening
access (since PERMISSIVE policies are OR'd). Intentional?

**8. Admin-vs-user split on `introductions`.**
There are 3 "Users can…" policies and 4 "admin_…" policies. Same
behavior could be expressed as 3 policies with
`USING: <user_condition> OR is_admin()`.

**9. `eco_applications` has no DELETE policy.**
An applicant withdrawing or admin removing a spam application can't
happen through the API. Intentional?

## Re-run these queries to re-audit

```sql
-- Table-level RLS status
select c.relname, c.relrowsecurity,
       (select count(*) from pg_policy p where p.polrelid = c.oid) as policy_count
from pg_class c
join pg_namespace n on n.oid = c.relnamespace
where n.nspname = 'public' and c.relkind = 'r'
order by c.relname;

-- Every policy, fully expanded
select c.relname as table_name,
       p.polname as policy_name,
       case p.polcmd when 'r' then 'SELECT' when 'a' then 'INSERT'
                     when 'w' then 'UPDATE' when 'd' then 'DELETE'
                     when '*' then 'ALL' end as command,
       pg_get_expr(p.polqual, p.polrelid)      as using_expr,
       pg_get_expr(p.polwithcheck, p.polrelid) as with_check_expr
from pg_policy p
join pg_class    c on c.oid = p.polrelid
join pg_namespace n on n.oid = c.relnamespace
where n.nspname = 'public'
order by c.relname, p.polname;
```
